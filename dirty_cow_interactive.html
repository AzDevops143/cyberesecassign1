S1
Stage 01 · Kernel Memory Exploit
Dirty COW — CVE-2016-5195
A race condition in the Linux kernel's Copy-On-Write mechanism allows an unprivileged attacker inside a container to overwrite any read-only file on the physical host disk, instantly gaining root privileges.
Attack Vector
Shared kernel → mmap() R/O file → Race condition → Root!
  • 1
    Attacker opens a read-only system file (e.g. /usr/bin/passwd) and maps it into virtual memory using mmap(PROT_READ, MAP_PRIVATE) — the kernel creates a safe private copy
  • 2
    Thread 1 calls madvise(MADV_DONTNEED) in a tight loop — continuously discarding the kernel's temporary copy of the mapping
  • 3
    Thread 2 writes a malicious payload through /proc/self/mem — targeting the same memory region
  • 4
    Race window hit: kernel's fault handler gets confused — the temporary copy was discarded by Thread 1 mid-write, so the kernel writes the payload directly to the physical read-only file on disk
  • 5
    Critical system file overwritten. Attacker now has root privileges inside the container — container boundary shattered at the kernel layer
Mitigation — Seccomp BPF
madvise() → Seccomp eBPF → EPERM → Race impossible
  • 1
    Interception layer: A custom hexaforce-seccomp.json profile is injected into the kernel's eBPF engine — acting as a high-speed firewall for syscalls
  • 2
    When Thread 1 attempts madvise(MADV_DONTNEED), the request hits the Seccomp filter before reaching the memory manager
  • 3
    Profile flags madvise as SCMP_ACT_ERRNO — the call is dropped and returns EPERM instantly
  • 4
    Thread 2's writes now only touch the private memory copy — the physical read-only file remains completely safe
  • 5
    Race condition physically cannot occur. Container boundary remains intact at the kernel level
Implementation Command
docker run \
  --security-opt seccomp=hexaforce-seccomp.json \
  -it vulnerable_image /bin/bash
S2
Stage 02 · Lateral Movement
CAP_SYS_PTRACE — Process Namespace Bypass
With root inside the container, the attacker abuses a shared PID namespace and the ptrace capability to scan host processes, inject shellcode, and escape the container entirely.
Attack Vector
--pid=host → Scan /proc → ptrace attach → Shellcode runs
  • 1
    Container was launched with --pid=host — the container shares the host's full PID namespace. Attacker can see and interact with every process on the bare-metal server via /proc
  • 2
    Attacker (now root from Stage 1) scans /proc to identify a high-value host process: orchestration agent, syslog daemon, cron
  • 3
    ptrace(PTRACE_ATTACH) is called on the target — the host process freezes instantly. CAP_SYS_PTRACE grants full control over its memory and CPU registers
  • 4
    Malicious shellcode is written into the target's memory via PTRACE_POKEDATA — overwriting legitimate code
  • 5
    PTRACE_CONT resumes execution. The host process now runs the attacker's shellcode natively on bare-metal, outside Docker/Kubernetes boundaries
Mitigation — Capability Dropping
Strict PID ns + ptrace() → LSM blocks → EPERM
  • 1
    PID namespace isolation: Docker's default strict PID namespace means the container only sees its own processes. Host processes are completely invisible — attacker's /proc scan finds nothing
  • 2
    --cap-drop=ALL strips every Linux capability from the container — including CAP_SYS_PTRACE — from the bounding set at launch time
  • 3
    When ptrace() is called, the Linux Security Module (LSM) intercepts and checks the capability bounding set — CAP_SYS_PTRACE is absent
  • 4
    Syscall is terminated with EPERM before any target process is touched. The attacker is a neutered root — technically "root" but without the kernel-level powers needed for injection
Implementation Command
docker run \
  --cap-drop=ALL \
  # Do NOT use --pid=host
  -it vulnerable_image /bin/bash
S3
Stage 03 · API Surface Attack
Docker Daemon API Hijacking via Socket
A mounted Docker socket gives the attacker unauthenticated HTTP access to the entire container engine running as root on the host — effectively handing over "God-mode" over the entire node.
Attack Vector
docker.sock mounted → curl unix-socket → Docker API (no auth) → Deploy containers
  • 1
    A DevOps convenience flag mounts the host's Docker socket: -v /var/run/docker.sock:/var/run/docker.sock — this socket is the literal control plane for the entire Docker daemon
  • 2
    Attacker discovers the socket file inside the container and recognizes it as the Docker control interface
  • 3
    The Docker socket speaks plain unauthenticated HTTP over a Unix domain socket — anyone who can write to it has full daemon access
  • 4
    Using curl --unix-socket /var/run/docker.sock, attacker sends JSON payloads to download images, delete production containers, or deploy rogue privileged ones
  • 5
    Docker engine (running as root) blindly obeys. Attacker controls the entire node's container ecosystem
Mitigation — Socket Isolation + mTLS
No socket mount → curl → sock → File not found
  • 1
    Primary fix — air-gap the control plane: Simply omit -v /var/run/docker.sock. The socket file does not exist inside the container. Mount namespace isolation makes the host's /var/run completely invisible
  • 2
    Attacker's curl commands return "Connection Refused" — the Docker engine is invisible and unreachable from inside the container
  • 3
    Fallback for CI/CD runners that need Docker API: configure daemon to listen on a TLS TCP port with Mutual TLS (mTLS) authentication
  • 4
    Container must present a valid signed client certificate. Attacker without private keys receives a hard 403 Forbidden — even if they find the API endpoint
Safe Launch — No Socket Mount
# NEVER do this for untrusted containers:
# -v /var/run/docker.sock:/var/run/docker.sock

docker run \
  --network=none \
  --cap-drop=ALL \
  -it vulnerable_image /bin/bash
S4
Stage 04 · Filesystem Persistence
Persistent Host Takeover via Filesystem Pivot
Using hijacked API access from Stage 3, the attacker spawns a privileged sibling container with the host's entire filesystem mounted inside it, then plants a permanent cron backdoor that survives container deletion.
Attack Vector
Spawn sibling container → -v /:/mnt/host → Write cron.d → Persistent shell
  • 1
    Using Docker API control from Stage 3, attacker commands the host daemon to spawn a new privileged sibling container next to the original one
  • 2
    The new container is created with -v /:/mnt/host — a bind mount that attaches the entire physical host filesystem inside the container at /mnt/host
  • 3
    Docker daemon runs as root and bypasses standard Linux file permissions — the disk is mounted with full read/write access
  • 4
    Attacker navigates to /mnt/host/etc/cron.d/ — which is actually the host's real system cron folder — and drops a reverse shell script file
  • 5
    Host's cron daemon wakes up, finds the new task, and executes it as root. Backdoor persists permanently — even if all containers are deleted and Docker is reinstalled
Mitigation — Read-Only Immutable FS
echo → cron.d → VFS intercepts → EROFS → Zero persistence
  • 1
    Immutable containers: The --read-only flag tells the kernel's Virtual File System (VFS) layer that nothing inside the container environment is allowed to be modified
  • 2
    When the attacker's echo command attempts to write to /mnt/host/etc/cron.d/, the VFS intercepts the write operation and sees the Read-Only flag
  • 3
    Write is terminated at the lowest kernel level with EROFS (Read-only file system). The malicious file is never created
  • 4
    Legitimate write needs (logs, temp files) are handled via --tmpfs /tmp — isolated in-memory volumes that never reach the host filesystem
  • 5
    Container restart wipes all in-memory state. Attacker is instantly evicted with zero persistence — host cron directory remains clean
Implementation Command
docker run \
  --read-only \
  --tmpfs /tmp \
  --cap-drop=ALL \
  -it vulnerable_image /bin/bash
